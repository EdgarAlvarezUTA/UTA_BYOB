| UTA expectation is that the URL will remain the same after the migration. Is it achievable? If so, how do we do it (e.g. what APIs to call)? | If the migration uses a new Cloudinary environment, the default delivery URLs change because the cloud name changes. However, if delivery uses a CNAME the externally visible URLs can remain the same by keeping the same CNAME and updating its DNS target to the new environment after migration. No API calls are required; the switch happens at the DNS/CNAME level. |
| You confirmed "the URLs will have to change because the environments are different." Can you provide a concrete before/after URL example for one of UTA's assets? Specifically: | Example using default Cloudinary delivery:<br>Before: https://res.cloudinary.com/old_cloud_name/image/upload/sample.jpg<br>After: https://res.cloudinary.com/new_cloud_name/image/upload/sample.jpg<br><br>If using a CNAME (for example https://media.example.com/image/upload/sample.jpg) the visible URL can remain the same and only the DNS target behind the CNAME changes. |
| Does the cloud name change? | **Yes. If a new environment is created the cloud name will change**. You can also change the existing cloud name to free the namespace and apply it to the new environment. |
| Does the public_id / asset path change? | **No. The public_id can remain the same as long as assets are migrated with the same identifiers**. This is the recommended approach so existing URLs continue to work. |
| Does the CNAME subdomain itself change, or does only its DNS target change? | **The CNAME itself does not have to change**; the DNS settings would remain the same on your side and we would update to the new cloud on our side. |
| UTA's applications likely reference Cloudinary URLs directly (e.g., in database records, CMS content, or hardcoded configs). If the URL structure changes, every reference in UTA's system needs to be remapped. Is there any migration path that preserves the existing URL structure end-to-end, or is a URL change unavoidable with the new-environment approach? | **The only way to preserve the externally visible URLs is to keep the same public_ids and continue delivering through the same CNAME.** If applications reference res.cloudinary.com/<cloud_name> directly, then the URL change is unavoidable because the cloud name differs between environments. |
|  Can we leverage a Cloudinary backup to client owned external S3 storage bucket as the source for migration? | **No. Cloudinary backup storage cannot be used as a migration source**. Backup buckets are managed internally for disaster recovery and are not exposed as a dataset for migration additionally the naming and folder structure differs significantly. |
| During the CNAME cutover window, you suggested using res.cloudinary.com delivery as a temporary fallback. <br>What is the expected duration of that window? </br><br>What happens to in-flight requests during the switch?</br>| **The cutover is typically very short** and depends mainly on DNS TTL settings. In most cases it is minutes, but it ultimately depends on DNS caching behavior. |
| Are there two simultaneous CNAMEs pointing to different environments, or is there a hard switch? | **Typically we would swap one CDN provider at a time**, and because we use multiple CDN providers, at one point there would be delivery from both clouds simultaneously from the same CNAME. |
| Will public_ids (asset identifiers) be preserved exactly as-is in the new environment, or can they differ? If they can differ, what is the mechanism to reconcile old vs. new public_ids? | **Yes**. Public IDs should be preserved during migration so that delivery URLs and transformations continue to work correctly. |
| What is it that Cloudinary Professional Services(PS) needs to do versus what we/client side can do? And if PS is needed, what is the PS engagement model (hours, timeline, cost)? | Professional Services typically handles BYOB backend configuration, S3 permission validation, and environment setup. **The customer or partner typically handles preparing the S3 bucket, executing the asset transfer**, and validating the application integration. |
| Does PS physically transfer original files from Cloudinary's S3 to UTA's S3, or do they only configure the backend pointer on Cloudinary's side? <br>If PS does not move the files, who is responsible for the actual data transfer (~3TB)? </br>| **PS generally does not transfer the files**. They configure and validate the backend setup. The actual data transfer is typically handled by the customer or partner using tools such as the Cloudinary CLI sync command, auto-upload or custom scripts.|
| After BYOB is configured, when a new asset is uploaded via the Cloudinary API or SDK, where does the original land: UTA's S3, Cloudinary's storage, or both? | Once BYOB is active, **new uploads via the Cloudinary Upload API or SDKs are stored in the customer’s S3** bucket as the primary storage location, while Cloudinary continues to manage metadata, transformations, and CDN delivery. |

While enterprise plans may support higher limits, **limits are still enforced per environment**, and temporary increases can be provided for migrations if needed.<br>Current Admin API responses show:**rate_limit_allowed: 5000**This means the environment currently has a 5,000 Admin API requests/hour limit. | You confirmed that existing backup history will be lost and deleted assets can no longer be recovered post-migration. Is there any window — before PS deletes the old environments — during which UTA can extract or archive the current backup data for their own retention? | **There is no supported method to export Cloudinary’s internal backup storage**. However, the old environments do not need to be deleted immediately after migration. They can remain active temporarily to allow validation and manual retrieval of assets if necessary before final decommissioning.|
