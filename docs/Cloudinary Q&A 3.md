| Can BYOB be enabled on UTA's existing Cloudinary environment (same cloud_name, same URLs), or does it always require a brand-new environment? The new environment lives within the same Cloudinary account but is fully segregated from the existing one. |

| cld sync is recommended for downloading assets — can you confirm the direction? Is it a pull (Cloudinary → destination) when run with a local path as target? What is the exact command flag for exporting all originals out of a Cloudinary environment? | Danny clarified that **`cld sync` synchronizes between a local folder and a Cloudinary folder** — it is a local-filesystem <-> Cloudinary operation, not an environment-to-environment operation. For environment-to-environment asset transfer, the correct tool is **`cld clone`**. Exact command flags for full-environment export should be confirmed with PS/documentation. |
| Does cld sync support an S3 bucket as the destination (e.g., s3://uta-migration-bucket/), or does it strictly require a local filesystem path as the intermediary? For environment-to-environment transfer, `cld clone` is the recommended approach (server-side, no intermediary machine required). Whether AWS-native tools (DataSync, S3 Batch) can be used as an alternative is deferred to the PS scoping session. |
| What does cld clone do exactly?<br>Does it copy binary files AND metadata between two Cloudinary environments, or metadata only?</br><br>Is the copy server-side (Cloudinary-to-Cloudinary), or does the binary data pass through the client machine?</br>Does it preserve public_id, tags, context fields, structured metadata, and transformation history? | Danny confirmed: **`cld clone` copies both binary files and associated metadata directly between two Cloudinary environments (server-side — data does not pass through the client machine).** It preserves `public_id`, tags, and contextual metadata. It supports optional parameters/switches to include or exclude tags and metadata. **Important exception:** structured metadata taxonomy is **not** copied by the clone command; it requires a separate bulk export → re-import process (see Q9). Upload presets and certain environment configuration items must also be set up independently after the clone. |
| Whether using cld sync or cld clone — is all metadata fully preserved (public_ids, tags, context, structured metadata, upload presets, transformations)? Or does metadata need to be migrated separately via the Admin API? | Metadata migration is a **staged process**: (1) `cld clone` preserves `public_id`, tags, and contextual metadata automatically. (2) **Structured metadata taxonomy is NOT copied** — it requires a bulk metadata export from the source environment, followed by a re-import/attachment to assets in the new environment after the clone completes. (3) **Upload presets and environment configuration** must be independently reconfigured on the new environment. Cloudinary has established processes for all of these steps and handles them as part of standard migration engagements. |
